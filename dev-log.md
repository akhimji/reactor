## 2026-05-01 — Phase 10 lands; sim core complete

The simulation core is done. Phase 10 — the event flush — landed as a `Sim` wrapper class in `packages/sim/src/sim.ts` that owns a type-keyed subscriber registry (`Map<EventType, Set<Handler>>`). `Sim.tick(inputs)` advances state via `advanceTick`, dispatches each event in `pendingEvents` to subscribers registered for that event type, then clears the queue. Subscriber errors are caught per-handler and logged via `console.error`, isolating one bad subscriber from the rest. The pure-functional core (`createSimState`, `advanceTick`) is unchanged — Sim is a thin convenience layer, and both interfaces remain exported. Phase 10's slot inside `advanceTick` stays a structural no-op, reserved for future in-tick concerns (filtering, batching, prioritization). Two new ADRs landed: ADR-034 (type-keyed subscription model — handlers register per event type, type narrowing preserved at the boundary, no wildcard in v1) and ADR-035 (subscriber errors are caught and continue dispatch). The `balance.ts` script gained a third `mode` arg (`raw` vs `sim`) demonstrating both interfaces and the tools README documents when to use which.

The integration test "full chain reaction observed via subscribers" exercises the renderer's-perspective path end-to-end: it drives a forced-split U235 chain through Sim using only public input commands (placeFuelRod → injectNeutron → scram), collects events via subscribers, and asserts the observed sequence matches a parallel ground-truth run through raw `advanceTick`. No events lost, ordering preserved, runEnded payload consistent with sim state. The test caught one design subtlety worth noting: Sim's public API takes only `(config, seed)` so seeding pre-built state for tests means either using fuel-rod placement (which I chose, exercising the public API faithfully) or punching through the private state field. The fuel-rod path is slower to set up than direct atom injection but it validates more of the renderer's actual entry path. Test count went 201 → 224 (+22 sim.test, +1 integration). `pnpm check` is fully green. Sim core is complete; the next session opens `packages/game` and starts wiring Phaser scenes against the subscriber API.

## 2026-05-01 — Phase 8 (criticality) lands; sim 9-of-10 phases complete

The reactor's central tension is now live in code. Phase 8 implements the rolling-window criticality computation per spec §5.1: each tick it sums `atomSplit.neutronsReleased` from `pendingEvents`, writes the slot at `tick % criticalityWindow` of `state.fissionHistory`, sums the array, and derives `k = windowTotal / (windowSeconds * baselineNeutronRate)`. Zone is classified per spec §5.2 boundary inclusivity (extinct → subcritical → nominal → supercritical → runaway → meltdown). Two events fire: `tick` every tick (with `criticality` and `zone` payload) and `criticalityZoneChanged` only on transitions. Score accumulates per ADR-032's quadratic edge-multiplier formula while in nominal — peak score at the boundaries (k=0.9 or k=1.1) is 2× base rate; center coasting earns 1× base rate. Five new ADRs landed: ADR-029 (naive recompute over circular buffer), ADR-030 (sum of `neutronsReleased`, not event count; atomDecayed excluded), ADR-031 (two-event emission strategy), ADR-032 (quadratic edge multiplier), ADR-033 (score lives at SimState top level). State gained `fissionHistory: readonly number[]` (length `criticalityWindow`), `score: number`, and the previously-optional `criticality` is now nullable required. Phase 1 (SCRAM) and phase 9 runEnded payloads now read `state.score` instead of hardcoding 0.

Wiring phase 8 surfaced a few interactions worth recording. (1) Phase 9 tests had been driving criticality directly via a `withCriticality` helper; with phase 8 active that helper is overwritten before phase 9 reads, so tests were rewritten to pre-populate `fissionHistory` in a slot phase 8 won't overwrite (helper `withTargetK`). The boundary test for k=0.1 was retargeted to k=0.125 because integer fission counts can't represent 1.6 neutrons exactly. (2) The pre-existing Pu239 timer-decay integration test (1799 ticks) now needs a custom config with extended `extinctionGracePeriod`: with phase 8 active, an empty playfield produces k=0 every tick and phase 9 extincts at tick 300, well before Pu239's 1800-tick decay timer. (3) Chain dynamics revealed a balance finding: with forced-split U235 (2 neutrons per split) in a tight grid, k can cross 0.9 → 1.1 in a single tick, skipping nominal entirely — meaning the player would never earn score on a chain that ramps too fast. The "sustained chain reaches nominal" integration test uses seed 7 specifically because it briefly visits nominal; seed 42 skips. Worth a tuning pass on `criticalityWindow` or per-atom yield when balance work begins. Test count went 157 → 201 (+41 phase 8, +3 criticality lifecycle integration, -1 obsolete phase-9-stub test, +1 lint cleanup nets to 44 net new tests). `pnpm check` is fully green.

## 2026-04-30 — Phases 7 (auto-decay) and 9 (end conditions) land

The sim now handles atom timeouts and run-end semantics. Phase 7 implements two paths per ADR-027 and ADR-028: Pu239 timer-driven decay (when `state.tick >= atom.decaysAt` and the atom is still `intact`) emits `atomDecayed` and removes the atom immediately, bypassing the `spent` state entirely; spent atom cleanup (when `state.tick - atom.spentAt >= spentAtomCleanupTicks`, default 60) silently removes the atom with no event since the prior `atomSpent`/`atomDecayed` already reported end-of-life. Atom gained `spentAt?: number` populated by phase 4 (absorb path, except B10) and phase 6 (splitting → spent). Phase 9 implements meltdown (`k > 2.0`, immediate end), extinction (`k < 0.1` for `extinctionGracePeriod` = 300 ticks, with rebound resetting the counter), and a Site objective stub. SimState gained `ticksBelowExtinction: number` (grace counter, resets to 0 on rebound) and `criticality?: SimCriticality` (optional pending phase 8). Three new ADRs landed: ADR-027 (spent atom cleanup mechanism), ADR-028 (atomSpent vs atomDecayed semantics — used vs timed out), and an implementation note on ADR-023 (phase 5 same-tick deferral guard).

One design tension surfaced during phase 9: the task's pre-decision said "read `state.criticality?.k ?? 0`," but defaulting k=0 when phase 8 hasn't yet populated criticality would silently extinct any run exceeding the grace period — including the new Pu239 decays-unused integration test (1800 ticks). Resolved by keeping `state.criticality` optional and gating both meltdown and extinction checks on its presence (mirrors the Site objective stub pattern); when phase 8 lands and populates criticality every tick, the checks engage automatically. The integration test for Pu239 decay passed cleanly on first run with no state-propagation issues. Test count went 117 → 157 (20 phase 7, 18 phase 9, 2 integration). `pnpm check` is fully green.

## 2026-04-29 — Phases 5 (apply collisions) and 6 (advance atom states) land

The atom state machine is wired end-to-end. Phase 5 reads `pendingNeutrons` off `excited` atoms whose `excitedSince < currentTick`, spawns that many neutrons via ADR-025's evenly-distributed-with-jitter algorithm (`N + 1` PRNG draws per split, single rotation seed plus per-neutron jitter inside `±jitterMagnitude/2`), positions each one just outside the parent's collision radius per ADR-026 (`collisionRadius + 0.01` epsilon to prevent next-tick re-collision), and transitions the atom to `splitting` with `splittingStartedAt = currentTick`. Phase 6 reads `splitting` atoms and transitions them to `spent` once `physics.splittingDuration` ticks (default 4) have elapsed since `splittingStartedAt`, emitting `atomSpent`. Two new ADRs documented the load-bearing decisions: ADR-025 (the angle algorithm and PRNG draw budget) and ADR-026 (same-tick deferral plus geometric offset for split-spawned neutrons). Spec §4.1.5/§4.1.6 were rewritten as proper subsections that resolve ownership unambiguously: phase 5 owns `excited → splitting` and neutron spawning, phase 6 owns `splitting → spent`. Atom gained `splittingStartedAt`; config gained a new `physics` category for `splittingDuration` and `neutronReleaseJitter`.

One pre-decision was tightened during implementation: phase 5 had to skip atoms with `excitedSince === currentTick` to preserve ADR-023's "1-tick excited window" — without this, phase 5 would transition the atom in the same tick that phase 4 set it to excited, collapsing the renderer's buildup-animation window to zero ticks. The session's planning notes implied "phase 5 owns the entire transition" without flagging this same-tick subtlety, so the spec amendment was extended to make the deferral explicit (same family of rule as ADR-016 for neutrons). The first true integration test (`integration.test.ts`) covers a U235 atom from injection through split, excited, splitting, and spent, with event-sequence and determinism assertions; it caught no regressions on first run, which validated the phase-4 / phase-5 / phase-6 wiring. Test count went 77 → 117: 23 phase-5 tests, 14 phase-6 tests, 3 integration tests. `pnpm check` is fully green.

## 2026-04-29 — Phase 4 (resolve collisions) lands

Phase 4 — the largest single chunk of sim logic in the project — is in. Six new ADRs encode the architectural decisions that drove it: ADR-019 (newly-spawned atoms are eligible for same-tick collisions, drawing the boundary against ADR-016), ADR-020 (losing neutrons in spawn-order pass through with no event; phase 4 holds a per-tick `hitAtoms` exclusion set), ADR-021 (control rod `absorbStrength` is per-tick Bernoulli, producing emergent speed-dependent absorption), ADR-022 (single-roll three-outcome split/absorb/pass-through with fixed evaluation order; B10 is immortal under absorption in v1), ADR-023 (atom state machine — `intact → excited` in phase 4, `excited → splitting` deferred to phase 6, `pendingNeutrons` field carries the count from phase 4 to phase 5/6), and ADR-024 (collision detection is O(n×m) brute force behind a clean `findNeutronAtomCollisions` interface, swap-in optimization deferred until profiling demands it).

Implementation lives in `packages/sim/src/collisions.ts` (swept-segment-vs-circle detection, ~70 lines) and `phaseResolveCollisions` in `tick.ts` (~150 lines). Atom gained `collisionRadius` (per-instance, populated from new `config.atom.collisionRadius` at spawn) and optional `pendingNeutrons` (set on split, consumed by phase 5). Default `controlRod.absorbStrength` was lowered from 0.7 to 0.4 to match the new per-tick semantics — placeholder pending balance work. All randomness threads through the seeded PRNG; one draw per resolved atom collision, plus one extra draw for the neutron count when the outcome is `split`, plus one draw per neutron-rod inside-radius check. Test count went from 37 → 77: 11 new collision-detection tests in `collisions.test.ts` and 29 new phase-4 tests in `phase4.test.ts` covering outcome resolution per atom type, spawn-order tiebreaks, state machine entries, same-tick spawning rules, control rod absorption, event payloads, determinism, and input immutability. `pnpm check` (typecheck + lint + test) is fully green.

Code migrated to match. `FuelRod.radius` is now on the entity; `applyPlaceFuelRod` populates it from `config.actions.fuelRod.radius` at placement, and `phaseAdvanceFuelRods` reads it per-rod for the candidate-radius check. Event payload emissions in phases 1, 2, 3 were updated to the new typed shapes (full `position`, `velocity`, `type`, `radius` fields where the spec calls for them; entity-disambiguated id field names like `atomId` / `neutronId` instead of bare `id`; `runEnded.outcome` with `finalTick` and `finalScore`; `RunEndReason` union member `'objective'` renamed to `'sustained'` to match the spec). ## 2026-04-28 — Phases 1+2 (process inputs, advance fuel rods) land

Phase 1 covers all four input commands: `injectNeutron`, `placeControlRod`, `placeFuelRod`, `scram`. Cooldowns (`{ injectNeutron }`) and inventory (`{ controlRods, fuelRods, scramAvailable }`) now live on `SimState`, sourced from new `actions.controlRod.inventoryDefault`, `actions.fuelRod.{ radius, releaseDuration, inventoryDefault }`, and `actions.scram.availableDefault` config fields. Invalid commands are dropped silently per spec §10. SCRAM sets `state.ended = 'stabilized'` and short-circuits subsequent inputs in the same tick. Phase 3 was extended by one line to skip neutrons whose `spawnedAt` equals the current tick — newly-spawned neutrons sit at the spawn position for one tick before moving, which honors the "phase 3 cleans up on the next tick" pre-decision and gives the renderer a stable spawn frame. Phase 2 walks each rod's `releaseSchedule`, releases atoms whose `atTick` matches the current tick, runs an 8-candidate spacing-aware offset adjustment (PRNG-seeded rotation for determinism), skips releases that have nowhere to land, and emits `fuelRodExhausted` exactly once when the rod transitions to exhausted. Spent rods stay in state for phase 7 cleanup later.

Spec amendments and ADRs cleaned up the contract while we were here. Sim spec §2.2 now says `expiresAt` means "lifetime end" and is never written to by other removal paths. §8.1 says `neutronExpired` carries `reason: 'expired' | 'out-of-bounds' | 'absorbed'` (the third one is forward-looking, for phase 4/5 collision absorption). ADR-013's tie-break reasoning was tightened from "integer-tick-deterministic" to "causal priority — a neutron at lifetime end is definitionally gone before any location check." ADR-014 makes event schema changes a spec amendment, not silent code drift; ADR-015 establishes that fields have one meaning and removals are operations, not field overloads. Test count went from 14 → 37 (15 phase-1 tests, 8 phase-2 tests). Phases 4–10 remain, with phase 4 (resolve collisions) being the next major piece — phases 5–10 are smaller in scope.
